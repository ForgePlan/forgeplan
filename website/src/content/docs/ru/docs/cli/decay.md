title: forgeplan decay
description: "Предварительный просмотр того, как предстоящие истечения срока действия доказательств повлияют на показатели R_eff по всему проекту."
---

`forgeplan decay` просматривает поля `valid_until` каждого EvidencePack и показывает, какие решения скоро потеряют R_eff. Помните: срок действия истёкшего доказательства не означает его исчезновение — оно ограничивается значением 0.1 (просроченное, но не отсутствующее). Поскольку R_eff является функцией «слабого звена», одно истекающее доказательство может перевести ранее «зелёное» решение в «красную» зону.

Это инструмент для планирования **Refresh sprint**: посмотрите, кто скоро станет просроченным, запланируйте `RefreshReport` и продлите доказательство, прежде чем оно даст о себе знать.

## Когда использовать

-   Начало месяца: «какие решения потребуют повторной верификации в этом спринте?»
-   Прежде чем полагаться на старое решение — не собирается ли его R_eff упасть?
-   После массового создания доказательств с одинаковым сроком действия — заметьте «обрыв» до того, как он наступит.
-   Начало сессии вместе с `forgeplan health` для полного обзора «что требует внимания».

## Когда НЕ использовать

-   Когда у вас вообще нет доказательств (R_eff уже 0) — нечему приходить в упадок.
-   Как одноразовый гейт качества — используйте `stale` для жёсткого обнаружения истечения срока действия.

## Использование

```text
forgeplan decay
```

## Опции

```text
  -h, --help     Print help
  -V, --version  Print version
```

## Примеры

### Просмотр предстоящего влияния распада

```bash
forgeplan decay
```

Вывод:

```text
Evidence decay report
─────────────────────
EVID-012  expires in  14d  PRD-003: R_eff 0.90 → 0.70
EVID-018  expires in  30d  ADR-005: R_eff 1.00 → 0.10  ⚠ cliff
EVID-021  expires in  45d  PRD-007: R_eff 0.80 → 0.60
EVID-030  expires in  60d  PRD-011: R_eff 0.90 → 0.80

4 evidence packs expiring within 60 days
2 decisions will drop below 0.7 (trust threshold)
```

### Включение в план спринта

Совместите со `stale` и `health`:

```bash
forgeplan decay          # что приближается
forgeplan stale          # что уже истекло
forgeplan health         # общий обзор слепых пятен
```

Эти три команды представляют полную картину «бэклога обновления».

## Интерпретация вывода

-   **expires in Xd** — дней до `valid_until`.
-   **R_eff X → Y** — текущий кэшированный R_eff против прогнозируемого после распада.
-   **⚠ cliff** — помечается, когда одно истечение срока действия переводит R_eff через границу доверия (0.9 → 0.1 — это «обрыв»; 0.9 → 0.8 — нет).
-   «Обрывы» — это приоритет: запланируйте `forgeplan new refresh "Re-verify EVID-XXX"` до указанной даты.

## Как это вписывается в рабочий процесс

```
monthly review → decay → plan refresh sprint → renew evidence → score
```

Распад является превентивным. Просроченность является реактивной. Запускайте `decay` до истечения срока действия доказательств; запускайте `stale`, чтобы поймать те, что вы пропустили.
