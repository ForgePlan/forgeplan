Показывает, как устаревшие/просроченные доказательства снизили оценки R_eff по всему рабочему пространству. Для каждого затронутого решения он сообщает **текущий** R_eff (с учётом штрафов за устаревание) по сравнению с гипотетическим **обновлённым** R_eff (если бы срок действия просроченных доказательств был продлён). Это позволяет количественно оценить «скрытую потерю качества», вызванную устареванием доказательств.

**Категория**: Качество

## Когда агент вызывает его

- **Аудит качества** — чтобы увидеть, какие решения незаметно теряют доверие из-за старых доказательств.
- **Приоритизация работы по обновлению** — наибольшие дельты между текущим и обновлённым R_eff указывают на обновления с наибольшим влиянием.
- **Перед релизом** — чтобы убедиться, что выпущенные решения не опираются на устаревшие доказательства.
- **Объяснение падения R_eff** пользователю («почему оценка PRD-039 изменилась с 0.85 на 0.41?»).

Устаревание следует правилу R_eff: просроченные доказательства не удаляются — они понижаются до CL0 с весом 0.1, поэтому самое слабое звено в цепи обрушивает общую оценку.

## Входные параметры

_Входные параметры отсутствуют. Вызовите этот инструмент с пустым объектом `{}`._

## Возвращает

```json
{
  "decayed": [
    {
      "id": "PRD-039",
      "kind": "prd",
      "r_eff_current": 0.41,
      "r_eff_fresh": 0.85,
      "delta": -0.44,
      "expired_evidence": [
        { "id": "EVID-047", "days_expired": 12 }
      ]
    }
  ],
  "total_impacted": 1
}
```

## Типичная последовательность

1. `forgeplan_stale` — найти просроченные артефакты.
2. `forgeplan_decay` — посмотреть, каким решениям они нанесли ущерб.
3. Отсортировать по `delta` по убыванию — сначала с наибольшим влиянием.
4. `forgeplan_renew` или создать новый `EvidencePack` → `forgeplan_link`.
5. `forgeplan_score` — подтвердить восстановление R_eff.

## Эквивалент CLI

```bash
forgeplan decay
```
