# Методология разработки — простым языком

> Как вести проект чтобы решения не терялись, код был подтверждён, а процесс не обходился.

---

## Главная идея

Каждая задача проходит 5 шагов:

```
Route → Shape → Code → Evidence → Done
```

- **Route** — определить масштаб задачи (мелкая или серьёзная?)
- **Shape** — описать что делаем и зачем (PRD/RFC)
- **Code** — написать код + тесты
- **Evidence** — подтвердить что работает (результаты тестов)
- **Done** — закрыть задачу, обновить прогресс

---

## Route: определи масштаб

Перед началом работы спроси себя: насколько это серьёзно?

| Масштаб | Что это | Что делать |
|---------|---------|-----------|
| **Tactical** | Мелочь: typo fix, одна строка | Просто делай, без документов |
| **Standard** | Фича на 1-3 дня | Опиши в PRD → напиши код → подтверди |
| **Deep** | Новый модуль, 1-2 недели | PRD → RFC (архитектура) → ADR (решения) → код |
| **Critical** | Кросс-команда, стратегия | Epic → PRD[] → RFC[] → ADR[] |

**Правило**: если сомневаешься — выбери уровень выше, не ниже.

---

## Shape: опиши что делаешь

### PRD — Product Requirements Document

Когда делаешь фичу, опиши:
- **Проблема** — что сейчас плохо?
- **Цели** — что должно стать лучше?
- **Не-цели** — что мы НЕ делаем
- **Требования** — конкретные пункты (FR-001, FR-002...)

### RFC — Request for Comments

Когда архитектура не очевидна:
- **Предложение** — как будем строить
- **Альтернативы** — что ещё рассматривали
- **Риски** — что может пойти не так

### ADR — Architecture Decision Record

Когда приняли важное решение:
- **Контекст** — почему возник вопрос
- **Решение** — что выбрали
- **Альтернативы** — что отклонили и почему

---

## Code: пиши с тестами

### Правило: каждая новая функция = тест сразу

Не переходи к следующей функции пока не написал тест для текущей. Не "потом добавлю" — сейчас.

### Чеклист перед коммитом

- [ ] Тесты написаны для нового кода
- [ ] Все тесты проходят (`cargo test` / `npm test`)
- [ ] 0 warnings
- [ ] Коммит ссылается на артефакт (PRD-001, RFC-002)

---

## Evidence: подтверди результат

После реализации — создай доказательство:
- Какие тесты прошли
- Какие метрики улучшились
- Что проверено вручную

Без evidence решение = "я так думаю". С evidence = "я проверил и вот результаты".

---

## 5 Hooks — автоматические проверки

| Hook | Когда срабатывает | Что проверяет |
|------|-------------------|---------------|
| **Safety** | Любая bash команда | Блокирует `git push --force`, `rm -rf` |
| **PR Todo** | Создание PR | Все P0 чекбоксы в TODO.md должны быть [x] |
| **Commit Test** | Коммит | Новые pub fn должны иметь тесты |
| **Pre-Code** | Редактирование кода | Должен существовать active PRD |
| **Pre-Commit Health** | Коммит | Предупреждает если есть blind spots |

Hooks не заменяют дисциплину — они ловят когда дисциплина ослабла.

---

## FPF — когда нужно подумать глубже

Используй `/fpf-simple` когда:
- Нужно разложить сложную задачу на части
- Нужно сравнить несколько вариантов решения
- Нужно оценить уверенность в решении
- Нужно найти что упущено

FPF — не бюрократия. Это способ думать структурно вместо "ну давай попробуем".

## Anti-patterns (что НЕ делать)

| Плохо | Почему | Вместо этого |
|-------|--------|-------------|
| Код без PRD (Standard+) | Непонятно зачем и что | Сначала опиши, потом пиши |
| Тесты "потом" | Потом = никогда | Тест сразу после функции |
| "Работает — значит готово" | Нет evidence = нет доверия | Подтверди результатами |
| Squash merge | Теряет поздние коммиты | Merge commit |
| `git add .` | Добавит секреты | `git add файл1 файл2` |
| Пропуск health на старте | Не видишь blind spots | `forgeplan health` первой командой |

## Шпаргалка: рабочий день

```
Утро:
  forgeplan health            ← где я? есть ли долги?

Новая задача:
  forgeplan route "описание"  ← какой масштаб?
  → Tactical: просто делай
  → Standard+: forgeplan new prd "Название"

Работа:
  написал функцию → написал тест → следующая функция

Закончил:
  cargo test                  ← все зелёные?
  forgeplan new evidence      ← подтверди результат
  git commit + push + PR      ← в ветку, merge commit

Вечер:
  forgeplan health            ← 0 blind spots? чисто?
```
