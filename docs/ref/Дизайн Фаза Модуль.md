# Модуль: FPF на этапе проектирования — практический гайд

**Для курса DEV VELOCITY**
**Источник:** реальная сессия онбординга в Claude Code с FPF Simple Skill

---

## 1. Когда нужен FPF: одно предложение

> **FPF нужен на этапе проектирования (Design Phase) — когда вы ещё не знаете, что именно делать, и цена ошибки выше цены паузы.**

Конкретнее, FPF покрывает всё «мышление перед действием»:

| Подэтап | Что решаем | Пример |
|---------|-----------|--------|
| **Discovery** | Понять, что вообще нужно делать | «У нас монолит на 500k строк — с чего начать?» |
| **Architecture** | Как это устроено | «PostgreSQL или MongoDB?» |
| **Planning** | Кто, что, в каком порядке | «5 команд, неясные требования — как организовать?» |
| **Decision-making** | Выбор из альтернатив (RFC, ADR) | «Делать свой auth или взять Auth0?» |
| **Team Design** | Роли, процессы, взаимодействие агентов и людей | «Как вести разработку в команде с агентами?» |

**Общий паттерн:** FPF полезен когда **думать сложнее, чем делать**. Если задача ясна и нужно просто написать код — FPF не нужен. Если непонятно *что* именно делать, *почему* или *как выбрать* — это территория FPF.

## 2. Что такое FPF (объяснение для студентов)

FPF (First Principles Framework) — это «операционная система для мышления». Не софт, не библиотека, а набор правил и паттернов, которые помогают думать строже и принимать решения лучше.

Представьте: вы стоите перед сложной задачей — много участников, противоречивые требования, неясно с чего начать. Обычно люди цепляются за первую идею и тянут её до конца. FPF заставляет вас систематически рассмотреть альтернативы перед тем, как выбрать.

### Ключевые идеи (без жаргона)

| Идея | Что значит простым языком |
|------|--------------------------|
| **Декомпозиция** | Разбей сложное на части, которые можно обсуждать по отдельности |
| **Bounded Contexts** | Каждая часть имеет чёткие границы — что внутри, что снаружи |
| **Роли** | Кто за что отвечает — не люди, а именно роли (один человек может играть несколько) |
| **Оценка уверенности** | Не просто «мы думаем так», а «мы думаем так, и вот насколько мы в этом уверены» |
| **Генерация альтернатив** | Прежде чем выбрать — систематически порождай варианты |

## 3. Как работает скилл технически

FPF Simple Skill — это агентский RAG без инфраструктуры.

```
Вы задаёте вопрос
    ↓
SKILL.md — роутер подбирает нужный раздел по «глаголу мышления»
    ↓
_index.md — оглавление раздела, агент выбирает подраздел
    ↓
Один файл (~300 строк) загружается в контекст
    ↓
Агент отвечает, используя FPF-паттерны
```

### Расход токенов

В контекст при старте загружается только SKILL.md — это **~108 строк**. Мизер.

56 000 строк спецификации лежат на диске и НЕ загружаются, пока не понадобятся. Когда агент решает, что нужен FPF:

1. Смотрит роутер в SKILL.md → находит нужный раздел
2. Читает `_index.md` (~20-30 строк) → выбирает подраздел
3. Загружает один файл ~300 строк

**Итого в худшем случае: ~108 + ~30 + ~300 = ~440 строк за один вопрос. Это очень мало.**

Если вы не задаёте вопросов, требующих FPF — скилл вообще не активируется.

## 4. Установка (1 минута)

```bash
npx skills add CodeAlive-AI/fpf-simple-skill -g
```

Флаг `-g` устанавливает глобально — скилл будет доступен во всех проектах.

После этого скилл подключится к Claude Code автоматически. Когда вы зададите вопрос, требующий глубокого анализа, агент сам активирует FPF и подтянет нужный раздел спецификации.

**Репозиторий:** https://github.com/CodeAlive-AI/fpf-simple-skill

## 5. 20 практических ситуаций для FPF

### Декомпозиция — разбить сложное на части

1. **«У нас монолит на 500k строк — как разбить на сервисы?»** — FPF даст bounded contexts, границы, критерии нарезки
2. **«Запускаем новый продукт — какие команды/роли нужны?»** — роли, ответственности, hand-offs между ними
3. **«Проект включает hardware, software и ML — как организовать работу?»** — декомпозиция по дисциплинам с сохранением целостности

### Принятие решений — выбрать из альтернатив

4. **«PostgreSQL или MongoDB для нашего кейса?»** — критерии сравнения ДО выбора, а не подгонка под ответ
5. **«Делать свой auth или взять Auth0?»** — систематическая оценка: стоимость, риски, зависимости
6. **«Монорепо или мультирепо?»** — FPF заставит определить «что значит лучше» прежде чем сравнивать

### Оценка уверенности — понять, насколько мы уверены

7. **«Архитектор говорит, что система выдержит 10k RPS — это надёжная оценка?»** — проверка обоснованности через evidence
8. **«У нас 3 противоречивых отчёта о причинах инцидента»** — оценка reliability каждого источника
9. **«Достаточно ли у нас данных, чтобы запускать эту фичу?»** — что мы знаем, чего не знаем, и как это проверить

### Генерация альтернатив — не застрять на первой идее

10. **«Как можно решить проблему задержек в пайплайне?»** — систематический перебор вместо «просто добавим кэш»
11. **«Какие есть способы масштабирования кроме горизонтального?»** — расширить пространство поиска
12. **«Мы всегда делаем REST API — а что ещё можно?»** — выход из привычного паттерна через альтернативы

### Разрешение конфликтов — договориться между сторонами

13. **«Backend хочет gRPC, frontend — REST, менеджмент — GraphQL»** — структурирование конфликта, поиск реальных потребностей за позициями
14. **«Security требует 2FA на каждый чих, UX говорит это убьёт конверсию»** — trade-offs, не «кто прав», а «как совместить»

### Унификация словаря — говорить на одном языке

15. **«Слово "сервис" значит разное для devops, backend и бизнеса»** — disambiguation, единый словарь
16. **«Новый человек в команде не понимает нашу терминологию»** — onboarding через bounded contexts

### Аудит и проверка

17. **«Наш техлид написал ADR — он хорошо обоснован?»** — проверка: есть ли evidence, рассмотрены ли альтернативы
18. **«Презентация для инвесторов — логика аргументации держится?»** — аудит цепочки рассуждений на ошибки

### Структурирование с нуля

19. **«Пишу RFC на новую систему — с чего начать?»** — шаблон: контексты → критерии → альтернативы → hand-offs → недостающие данные
20. **«Нужно провести design review — по каким осям оценивать?»** — характеристики системы, measurement framework

## 6. Задание для курса: Определи свои 5 кейсов

### Инструкция для студента

1. Прочитайте 20 ситуаций выше
2. Из ВАШЕЙ текущей работы определите **5 ситуаций**, которые подходят под один из паттернов (декомпозиция, выбор, оценка, генерация альтернатив, конфликт, унификация, аудит, структурирование)
3. Для каждой ситуации:
   - Опишите задачу в 2-3 предложениях
   - Определите тип мышления (какой из 8 паттернов)
   - Сформулируйте промпт для агента с FPF
4. Запустите 2 из 5 через Claude Code с FPF Skill
5. Сравните результат с тем, что бы вы получили без FPF

### Формат сдачи

Таблица из 5 строк:

| Ситуация | Тип мышления | Промпт | Результат с FPF (кратко) | Было бы без FPF |
|----------|-------------|--------|--------------------------|-----------------|
| ... | ... | ... | ... | ... |

### Критерии оценки

| Критерий | Отлично (5) | Хорошо (4) | Удовл. (3) |
|----------|-------------|------------|------------|
| Релевантность ситуаций | Все 5 из реальной работы, разные типы | 3-4 из реальной работы | Абстрактные примеры |
| Качество промптов | Конкретные, с контекстом проекта | Общие, но понятные | Размытые |
| Анализ разницы | Конкретные наблюдения с примерами | Общие наблюдения | Формальное «лучше/хуже» |

## 7. С чего начать читать FPF (для любопытных)

Если хочется погрузиться глубже — вот порядок чтения через агента (не сами файлы, а через диалог):

1. **Предисловие** — зачем вообще нужна архитектура мышления
2. **Часть A (Ядро)** — базовые кирпичики: холоны (части-целые), роли, методы
3. **Часть B (Рассуждение)** — как собирать части обратно и оценивать уверенность

Промпт для изучения:

```
У тебя есть FPF. Загрузи [Часть A / Часть B / Предисловие] 
и объясни мне ключевые концепции простым языком. 
Без формализмов, с примерами из разработки.
```

## 8. Место в конвейере разработки

```
┌─────────────────────────────────────────────────────────┐
│                    DESIGN PHASE                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐  │
│  │Discovery │→ │Architecture│→│Planning │→ │Decision│  │
│  │  Что?    │  │   Как?    │  │  Кто?   │  │ Выбор  │  │
│  └──────────┘  └──────────┘  └──────────┘  └────────┘  │
│                                                         │
│  ← FPF РАБОТАЕТ ЗДЕСЬ →                                │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│                 SPEC PHASE                               │
│  RFC, ADR, Epic, Feature Files, Спецификации            │
│  ← BMAT / Open Spec / Plan Mode →                      │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│                 CODE PHASE                               │
│  Реализация по спецификации                             │
│  ← Claude Code / Codex / Cursor →                      │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│                 VERIFY PHASE                             │
│  Тесты, ревью, приёмка                                  │
│  ← CI/CD + ручная проверка →                           │
└─────────────────────────────────────────────────────────┘
```

**Ключевой вывод:** FPF — это не замена кодинга, а то, что стоит ПЕРЕД ним. Чем лучше отработан Design Phase, тем меньше переделок на Code Phase.
