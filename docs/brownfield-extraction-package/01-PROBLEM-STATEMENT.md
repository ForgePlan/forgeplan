# Problem Statement

## The core problem

Brownfield business documentation generated by combining **code inventory tools** (like `autoresearch:learn`) with **artifact-lifecycle tools** (like `forgeplan`) produces documentation that is:

1. **Not standalone** — heavily depends on `file:line` references.
2. **Not intent-aware** — describes *what* the code does, not *why* the business needs it.
3. **Not reproduction-ready** — insufficient to rebuild the system from docs alone.
4. **Not validated** — no distinction between facts from code vs hypotheses about meaning.
5. **Not RAG-ready** — no stable chunk boundaries, no semantic tagging, no embedding-compatible metadata.

## Concrete gap analysis

### Gap 1: References instead of content

**Current**:
```markdown
Order has 9 statuses (see `models/Order.js:82-91`):
  created, accepted, cargo_owner_confirmed, forwarder_confirmed, ...
```

**Needed**:
```markdown
## Order.status enum (canonical definition)

An Order in TripSales progresses through the following lifecycle states.
Each state represents a business commitment by a specific actor:

| Status | Business meaning | Actor commits | Next allowed |
|---|---|---|---|
| created | Order exists but no party has committed | (none — system) | cargo_owner_confirmed, forwarder_confirmed, canceled |
| accepted | Customer accepted the quote that created this order | system (from quote acceptance) | forwarder_confirmed |
| cargo_owner_confirmed | Cargo owner finalized cargo specs and shipping addresses | cargo_owner employee | forwarder_confirmed (if forwarder hasn't confirmed yet) |
| forwarder_confirmed | Carrier committed transport capacity | forwarder employee | fulfillment (on trip start), canceled |
| fulfillment | At least one trip carrying this order has started | system (trip start) | completed |
| completed | All cargo delivered and confirmed | system (last trip complete OR warehouse.issue.cargo) | (paid flag can still flip) |
| canceled | Order terminated | any party with access | (terminal) |
| rejected | Defined in enum but NEVER set in code. [HYPOTHESIS: intended for quote rejection before order creation, migrated from older model. VERIFY with Domain Owner.] |
| removed | Defined in enum but NEVER set. [HYPOTHESIS: soft-delete marker paralleling `deleted_at`. VERIFY.] |

### DDL (PostgreSQL, standalone)

\`\`\`sql
CREATE TYPE order_status AS ENUM (
  'created', 'accepted', 'cargo_owner_confirmed', 'forwarder_confirmed',
  'fulfillment', 'rejected', 'canceled', 'completed', 'removed'
);

CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  status order_status NOT NULL,
  -- ... full schema ...
);
\`\`\`
```

**The difference**: the second is readable, understandable, and usable even if the codebase is deleted.

### Gap 2: No intent layer

**Current**: "`_confirm` branches on `user.company.type_company` — if forwarder → status = forwarder_confirmed; else → cargo_owner_confirmed".

**Needed** (intent tier added):
```markdown
## Business rule: dual-confirmation pattern

**Intent**: TripSales operates between two parties — a forwarder (carrier) and a cargo_owner (shipper). An order represents a commercial agreement that BOTH parties must validate before execution begins.

**Rationale** [HYPOTHESIS, confidence: medium]:
- Forwarders commit physical transport capacity (vehicles, drivers, routes).
- Cargo owners commit the actual shipment details (weight, volume, pickup address).
- If either side hasn't confirmed, the order is ambiguous and cannot proceed to fulfillment.

**Observable behavior from code**: (see factum section)

**Business impact if removed**: orders could start fulfillment with incomplete specs → wrong cargo loaded → financial loss.

**Open questions for Domain Owner**:
1. Is there an SLA on how long an order can wait between first-side confirm and second-side confirm?
2. What happens if only one side confirms and then cancels?
3. Can an order be "partially confirmed" in the UI or is it binary?
```

### Gap 3: No scenarios

**Current**: "`v4.sales.make.payment` action accepts `{sales_order_id, invoice_id, amount}`, calls `v4.payments.makePayment`, updates `paid`/`status`".

**Needed** (scenario):
```markdown
## Scenario: Partial payment of invoice

\`\`\`gherkin
Feature: Partial payment

  Background:
    Given an invoice I1 with total=1000 and status='invoiced'
    And a sales_order S1 with this invoice attached

  Scenario: First partial payment
    When the forwarder calls sales_MakePayment(S1, I1, 400)
    Then payment P1 is created with amount=400
    And invoice I1 paid=400, status='partially_paid'
    And sales_order S1 paid=400, status='partially_paid'
    And the underlying order.paid remains false

  Scenario: Second payment completing the invoice
    Given the first partial payment happened
    When the forwarder calls sales_MakePayment(S1, I1, 600)
    Then payment P2 is created with amount=600
    And invoice I1 paid=1000, status='paid'
    And sales_order S1 status='paid', paid_at=now()
    And the underlying order.paid is set to true
    And if sales_order.type='trip', trip.paid is set to true instead
\`\`\`
```

### Gap 4: No hypothesis scoring

**Current**: hypotheses about code behavior are mixed with facts without distinction.

**Needed**: every non-trivial claim tagged with confidence level:
- `verified` — confirmed by Domain Owner OR inevitable from code.
- `inferred` — reasonable interpretation of code but not confirmed.
- `speculation` — guess based on naming or patterns.

### Gap 5: No RAG-ready output

**Current**: markdown files with variable structure, no stable chunk ids, no metadata.

**Needed**: each artifact has:
- Stable `artifact_id` (survives renames).
- `chunk_id` per logical section (for embedding).
- `topic_tags` + `domain_tags` + `entity_refs` in frontmatter.
- Cross-references as graph edges (machine-readable).

## Consequences of the gap (what can't be done today)

Without fixing this gap, the documentation **cannot be used to**:

1. **Rewrite the project** in another language (Go, Rust, Python). Engineers would still need the original code — defeating the purpose of docs.
2. **Onboard a new team in isolation**. New developers would need access to the codebase to understand anything non-trivial.
3. **Ingest into a RAG** for an AI assistant. The chunks are file-path-dependent and lose meaning outside the project.
4. **Survive code reorganization**. A simple directory restructure breaks all references.
5. **Audit business invariants independently**. Auditors can't verify rules without reading the code.
6. **Generate test scenarios automatically**. No Gherkin-style specs to derive tests from.

## Who feels this pain

- **Engineers joining an existing team** — forced to learn through code archaeology.
- **Product managers** — cannot reason about the system at business level.
- **Domain experts reviewing proposed changes** — can't validate without the developer translating.
- **AI agents working on the codebase** — hallucinate because context is insufficient.
- **Auditors / compliance teams** — no self-contained source of truth.
- **The business itself** — all institutional knowledge lives in code + developers' heads.

## Goals of the extraction process (what we're building)

By the end of the extraction process, a brownfield project should have:

1. **Ubiquitous Language glossary** — 100-500 terms with definitions + synonyms + confidence.
2. **Use Case catalog** — all user journeys from entry points to business outcomes.
3. **Invariant catalog** — business rules extracted from code guards, grouped semantically.
4. **Scenario catalog** — Gherkin features for every non-trivial operation.
5. **Domain Models** — DDD-style aggregates with canonical definitions.
6. **Hypothesis register** — every uncertain claim tagged with confidence, verification path, owner.
7. **Standalone docs** — DDL, SDL, pseudo-code without file:line deps.
8. **Knowledge graph** — semantic links between all of the above.
9. **RAG package** — export-ready chunks with metadata.
10. **Interview packets** — structured Q&A sheets for Domain Owner.

## Scope of this design package

We specify **how to build the tools** that produce the above. We do NOT run them. The forgeplan maintainer agent builds the tools; a user runs them.

## Measurable success criteria

The package is successful if:

1. After implementing the skills + kinds, a brownfield project can be analyzed with one command (`/extract-business-logic`).
2. Output contains all 10 goal artifacts above.
3. A fresh AI agent (no prior code access) can answer "How does feature X work?" using ONLY the output docs with 80%+ accuracy on validation questions.
4. Output passes the "code deletion test": delete the codebase, can we still understand what the system does? YES.
5. A RAG ingestion of the output is queryable for business questions ("What happens when a quote is accepted?") with relevant retrieval.
6. Output can serve as a rewrite spec for a target language/stack with reasonable fidelity.
